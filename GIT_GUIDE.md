# Git 使用指南

## 一、Git 是什么？对你有什么用？

Git 是**版本控制系统**，核心作用是：

| 场景 | 没有 Git | 有 Git |
|------|---------|--------|
| 代码改坏了想回退 | 没救，手动改回来 | 一条命令回到任意历史版本 |
| 多人同时改同一个项目 | 互相覆盖，冲突混乱 | 自动合并，有冲突会提示你解决 |
| 想试试新方案但怕搞坏现有代码 | 不敢改，怕回不来 | 开个分支随便试，不行就删掉 |
| 想知道谁什么时候改了什么 | 全靠记忆和聊天记录 | 每次改动都有记录，可追溯 |

**一句话总结**：Git 给你的项目装了"时光机 + 协作工具"。

---

## 二、核心概念（5 个必须理解的）

```
工作区(你编辑的文件) → 暂存区(准备提交的文件) → 本地仓库(提交历史) → 远程仓库(GitHub)
```

- **工作区**：你实际编辑的文件夹
- **暂存区**：你选中的、准备打包提交的改动（用 `git add` 放入）
- **本地仓库**：你电脑上的提交历史（用 `git commit` 保存）
- **远程仓库**：GitHub 上的备份，供多人协作（用 `git push` 上传）
- **分支**：从主线分出来的独立开发线，互不干扰

## 三、日常使用流程（最常见的工作循环）

### 1. 修改代码后，提交到本地

```bash
# 查看哪些文件被修改了
git status

# 查看具体修改了什么内容
git diff

# 把修改的文件加入暂存区（挑选要提交的）
git add 文件名          # 提交指定文件
git add .              # 提交所有修改

# 提交到本地仓库，写清楚改了什么
git commit -m "修复了登录页面的bug"
```

### 2. 推送到 GitHub（备份 + 共享）

```bash
git push
```

### 3. 从 GitHub 拉取别人的更新

```bash
git pull
```

## 四、常用命令速查表

### 基础操作

| 命令 | 作用 | 使用场景 |
|------|------|---------|
| `git status` | 查看当前状态 | **每次操作前都看一眼**，养成习惯 |
| `git add .` | 暂存所有修改 | 改完代码准备提交 |
| `git commit -m "说明"` | 提交到本地 | 每完成一个功能/修复就提交一次 |
| `git push` | 推送到 GitHub | 提交后推上去备份 |
| `git pull` | 拉取远程更新 | 别人改了代码，你同步下来 |
| `git log --oneline` | 查看提交历史 | 看之前都改了什么 |

### 回退与撤销

| 命令 | 作用 | 风险等级 |
|------|------|---------|
| `git checkout -- 文件名` | 撤销工作区的修改（还没 add） | ⚠️ 修改会丢失 |
| `git reset HEAD 文件名` | 取消暂存（add 了但想拿出来） | 安全，文件还在 |
| `git reset --soft HEAD~1` | 撤销最近一次提交，保留修改 | 安全，改动回到暂存区 |
| `git reset --hard HEAD~1` | 撤销最近一次提交，**丢弃所有修改** | ❌ 危险，数据会丢 |

### 分支操作

| 命令 | 作用 |
|------|------|
| `git branch` | 查看所有分支 |
| `git branch 分支名` | 创建新分支 |
| `git checkout 分支名` | 切换分支 |
| `git checkout -b 分支名` | 创建并切换到新分支（常用） |
| `git merge 分支名` | 把指定分支合并到当前分支 |
| `git branch -d 分支名` | 删除分支 |

## 五、实际工作示例

### 场景 1：日常开发

```bash
# 1. 开始工作前，先拉取最新代码
git pull

# 2. 写代码...写代码...写完了

# 3. 查看改了什么
git status
git diff

# 4. 提交
git add .
git commit -m "添加了用户注册功能"

# 5. 推送到 GitHub
git push
```

### 场景 2：想试试新方案，怕搞坏现有代码

```bash
# 1. 创建一个实验分支
git checkout -b experiment

# 2. 在实验分支上随便改
# ...修改代码...

git add .
git commit -m "试验新的缓存方案"

# 3. 如果方案好用，合并回主线
git checkout main
git merge experiment

# 4. 如果方案不行，直接切回去，实验分支可以删掉
git checkout main
git branch -d experiment
```

### 场景 3：代码改坏了，想回退

```bash
# 先看提交历史，找到想回退的版本
git log --oneline

# 输出类似：
# a1b2c3d 添加了用户注册功能
# e4f5g6h 修复了登录bug
# 030e027 Initial commit: add README.md

# 回退到指定版本（保留修改在工作区）
git reset --soft e4f5g6h
```

## 六、最佳习惯

1. **频繁提交**：每完成一个小功能就 commit，不要攒一大堆
2. **写好提交信息**：`git commit -m "修复登录页面密码验证bug"` 比 `git commit -m "fix"` 有用得多
3. **经常 push**：相当于备份，电脑坏了也不怕
4. **经常 pull**：和队友协作时，每次开始工作前先拉取最新代码
5. **不确定就开分支**：任何有风险的改动都在分支上做，主线永远保持可用
